# CloudCampus V2.1 — Implementation Plan

**Baseline:** the WIP V2 branch (school-year/registration/announcements/forgot-password scaffolding already in working tree, see `Docs/V2_Implementation_Plan.md`).
**Target:** member-driven editing, finalized approval workflow, multi-incumbent officers, dropdown lookups, unique identity, QOL polish.

This is V2.1 — it presumes V2's tables exist (school_years, registration_requests, announcements, etc.). Each section is independent enough to ship on its own; sections are listed in dependency order.

## Phase 0 — Database changes (one migration: `0003_v2_1_editing_and_approval.sql`)

All schema changes are bundled in a single new migration so the API/UI work can land afterward without further DB round-trips.

### 0.1 Multi-incumbent officer positions

Drop the V2 singleton constraint in favour of an integer cap, and add a slot count to each position.

```sql
ALTER TABLE officer_positions
  ADD COLUMN max_incumbents INTEGER NOT NULL DEFAULT 1
    CHECK (max_incumbents >= 1);

UPDATE officer_positions SET max_incumbents = 1
 WHERE is_singleton;

-- Drop the V2 singleton index and trigger now that max_incumbents replaces them.
DROP INDEX IF EXISTS officers_singleton_per_sy;
ALTER TABLE officer_positions DROP COLUMN IF EXISTS is_singleton;

-- Enforce the cap (defense-in-depth; API also pre-checks).
CREATE OR REPLACE FUNCTION enforce_officer_cap() RETURNS trigger AS $$
DECLARE
  cap INT;
  used INT;
BEGIN
  SELECT max_incumbents INTO cap FROM officer_positions WHERE id = NEW.position_id;
  SELECT count(*) INTO used FROM officers
    WHERE position_id = NEW.position_id
      AND school_year_id = NEW.school_year_id
      AND is_current
      AND id <> NEW.id;
  IF used >= cap THEN
    RAISE EXCEPTION 'position % is at capacity (% of %)', NEW.position_id, used+1, cap;
  END IF;
  RETURN NEW;
END $$ LANGUAGE plpgsql;

CREATE TRIGGER officers_cap_check
  BEFORE INSERT OR UPDATE ON officers
  FOR EACH ROW WHEN (NEW.is_current) EXECUTE FUNCTION enforce_officer_cap();
```

### 0.2 Event approval rework

Replace V1's "all three positions must approve" trigger with a 2/3 majority gate and add a `revisions_requested` status.

```sql
ALTER TYPE event_status ADD VALUE 'revision_requested' BEFORE 'rejected';
ALTER TYPE event_status ADD VALUE 'revised'            BEFORE 'rejected'; -- editor pushed an update after revision_requested

-- The new threshold: ceil(2 * total_approvers / 3) approvals required, minimum 2.
-- Total approvers = the count of officer rows currently marked as approver positions.
CREATE OR REPLACE FUNCTION event_threshold(total INT) RETURNS INT AS $$
  SELECT GREATEST(2, CEIL(2.0 * total / 3.0)::INT);
$$ LANGUAGE sql IMMUTABLE;

-- New trigger replaces the old "advance to approved when 3 approve" logic.
-- See lib/queries.ts for the in-app mirror that pre-checks before insert.
DROP TRIGGER IF EXISTS event_vote_advance ON event_votes;

CREATE OR REPLACE FUNCTION event_vote_advance() RETURNS trigger AS $$
DECLARE
  approvers INT;
  approvals INT;
  rejections INT;
  needed INT;
  sy UUID;
BEGIN
  SELECT school_year_id INTO sy FROM events WHERE id = NEW.event_id;
  -- Approver pool: officers in the current SY whose position is_approver.
  SELECT count(*) INTO approvers FROM officers o
    JOIN officer_positions p ON p.id = o.position_id
   WHERE o.is_current AND o.school_year_id = sy AND p.is_approver;
  needed := event_threshold(approvers);

  SELECT count(*) INTO approvals FROM event_votes WHERE event_id = NEW.event_id AND decision = 'approved';
  SELECT count(*) INTO rejections FROM event_votes WHERE event_id = NEW.event_id AND decision = 'rejected';

  IF approvals >= needed THEN
    UPDATE events SET status = 'approved', approved_at = now() WHERE id = NEW.event_id;
  ELSIF rejections > (approvers - needed) THEN
    -- Enough approvers said no that approval is mathematically impossible.
    UPDATE events SET status = 'rejected' WHERE id = NEW.event_id;
  END IF;
  RETURN NEW;
END $$ LANGUAGE plpgsql;

CREATE TRIGGER event_vote_advance
  AFTER INSERT ON event_votes
  FOR EACH ROW EXECUTE FUNCTION event_vote_advance();
```

`event_votes.decision` gains `'revision_requested'`. When that vote is cast, the event flips to `revision_requested` and existing votes are cleared (re-vote required after the author resubmits).

### 0.4 Unique identity guards

```sql
-- Users.email already unique (CITEXT). Add the same on registration_requests
-- when status <> 'rejected' (partial unique index so rejected emails can re-apply).
CREATE UNIQUE INDEX registration_requests_email_active
  ON registration_requests (lower(email))
  WHERE status <> 'rejected';

-- registration_requests, AND mutually exclusive (can't reuse a member's student ID

CREATE UNIQUE INDEX registration_requests_student_id_active
  ON registration_requests (student_id) WHERE student_id IS NOT NULL AND status <> 'rejected';
```

## Phase 1 — Edit + re-approval (blogs, events, projects)

### 1.1 Authorship + edit guards

Add a helper `canEdit(session, item)` that returns true when:
- `session.memberId === item.author_id`, OR
- `session.role === 'admin'`

And the item is not in a terminal state (rejected items cannot be edited; the author must create a new draft).

Wire into all three areas: blogs, events, projects.

### 1.2 Edit pages

| Path | What |
|---|---|
| `app/(public)/blogs/[slug]/edit/page.tsx` + `edit-form.tsx` | Same fields as `/blogs/new/blog-form.tsx`, prefilled. PATCH `/api/blogs/[id]`. |
| `app/(public)/events/[slug]/edit/page.tsx` + `edit-form.tsx` | Same as new. PATCH `/api/events/[id]`. |
| `app/(public)/projects/[id]/edit/page.tsx` + `edit-form.tsx` | Same as new. PATCH `/api/projects/[id]`. |

### 1.3 API: PATCH endpoints

For each entity:

- `PATCH /api/blogs/[id]` — body matches the create payload. On any field change the route sets `status='pending'`, `edited_at=now()`, and (if the row was previously approved) preserves `previous_published_at = published_at`. Clears `approved_at`/`approved_by`. New cover/attachment uploads go through the deferred-upload flow we just shipped.
- `PATCH /api/events/[id]` — same, but only allowed when status ∈ {`pending`, `revision_requested`}. Moving from `revision_requested → revised` triggers a new vote round (existing votes were already cleared on the revision request).
- `PATCH /api/projects/[id]` — same as blogs.

### 1.4 Public visibility

- `/blogs`, `/events`, `/projects` (public pages) — list only items with `status='approved'`.
- Each page gets a "Your drafts" section visible only when the viewer is signed-in and authored anything. Shows pending / revision_requested / rejected items with the appropriate badges.

## Phase 4 — Email change with verification

Account email is identity-bearing — letting a logged-in user change it without verification opens account-takeover patterns.

Flow:

1. User clicks **Change email** on `/profile`.
2. Inline form takes the new email + current password (re-auth gate).
3. POST `/api/profile/email/initiate` → inserts row in `email_change_requests` (new table) with token hash + expiry; queues an email to the NEW address with a confirmation link.
4. Link → `/profile/change-email/confirm?token=…` → calls `/api/profile/email/confirm` → swaps `users.email` and marks the request used.

**Contact email** (the `members.contact_email` field, shown on the member's public profile) is just data — bind it to a normal text input with a "Save" button, no verification. The "use account email for contact email" button autofills the contact email from the current account email.

New table:

```sql
CREATE TABLE email_change_requests (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  new_email   CITEXT NOT NULL,
  token_hash  TEXT NOT NULL,
  expires_at  TIMESTAMPTZ NOT NULL,
  used_at     TIMESTAMPTZ,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

This depends on email delivery; if SES has not been re-enabled (we just reverted notifications), the form copy says so and disables the "Send confirmation" button.

## Open decisions

These are surfaced for the user to lock down before implementation starts:

1. **Approval revote on revision** — when an officer requests revision, do we keep their `request_revision` vote and only clear `approve`/`reject` from the others, or wipe ALL votes including the revision request? (Plan currently wipes all; cleaner UX but discards the "ask why".)
2. **Event approver pool** — the new threshold uses "officers in the current SY whose position is_approver". V1 hardcoded three position rows. V2.1's flexible cap means an SY could have e.g. 5 approvers (President + VP + Secretary + Treasurer + Auditor). Confirm the SRS doesn't pin this at 3.
3. **`projects.status`** — does V2 already add this column? If yes, Phase 0.7 reduces to just `published_url`.
4. **Email change confirmation** — Phase 4 requires SES delivery. We just reverted SES. Confirm whether Phase 4 ships with a "feature disabled" notice or you'll re-enable SES first.
5. **Rejected blogs/events/projects** — explicitly "creator cannot edit" per your spec. Should an admin be able to override and reopen for editing, or is rejection final?

## Out of scope for V2.1

- WebSocket-driven approval status (still requires a page refresh).
- Public RSVP / event registration (separate flow).
- Bulk import / CSV of members.
- Anything notification/email related beyond the email-change confirmation in Phase 4.

## Verification checklist

- Migration runs cleanly: `npm run db:migrate && npm run db:check`.
- A member can edit their own approved blog → it returns to pending and an officer sees it in the queue.
- An officer can approve a blog from the blog detail page without leaving for the admin queue.
- A position with `max_incumbents = 3` accepts three officers, rejects the fourth.
- An event with 5 approvers approves at 4 votes; rejects mathematically at 4 reject votes (2/3 = 4, 5-4 = 1, so once `reject_count > 1` and only 1 voter remains, approval is impossible).
- Course is selected from a dropdown; submitting a free-text value is rejected.
- Email-change confirmation flow sends a verification link to the new address (or returns a "SES not configured" error in a deterministic way).
- Two registration requests with the same student ID — the second fails with a unique-constraint error.
- Announcements forms only show a date picker, no time-of-day.
